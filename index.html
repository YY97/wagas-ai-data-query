🔵 阶段一

OpenClaw 小龙虾 — LLM + Agent 能力验证

利用 OpenClaw（小龙虾）平台，导入 Excel 业务样例数据，初步验证大模型与 Agent 在企业问数场景中的可行性。

▸ 方案
  • 通过 OpenClaw 平台能力导入 Excel 业务样例数据
  • 测试大模型对表格数据的理解与问答能力
  • 验证 Agent 调用链路的基本可行性
▸ 踩坑
  • 数据对接困难：Excel 离线导入，无法实时同步业务数据
  • 数据安全风险：样例数据上传到第三方平台，存在泄露隐患
  • 无法规模化：仅能处理静态快照数据，不支持动态查询
▸ 结论
  • LLM + Agent 路线技术上可行，但数据通道是核心瓶颈
  • 企业级落地需要解决数据安全与实时连接两大问题
🟢 阶段二

观远BI ChatBI — 商业产品试用

现有 BI 厂商观远数据提供 ChatBI 产品试用，评估商业产品是否能满足企业问数需求。

▸ 方案
  • 观远BI 开通 ChatBI 模块试用
  • 基于已有 BI 数据集进行自然语言问答
▸ 踩坑
  • 输出文件行数限制：查询结果超过一定行数被截断
  • 不支持文件上传：无法基于临时 Excel/CSV 提问
  • 无历史关联：不支持关联历史提问做上下文推理
  • 价格较贵：商业授权费用超出预算预期
▸ 结论
  • 功能基本可用，但限制较多且成本偏高
  • 需要更开放、更灵活的方案
🟡 阶段三

BI CLI + Agent → 钉钉机器人（CRM 场景）

BI 开放 CLI 能力后，通过 Agent 产品（Qoderwork）连接到 BI 数据集，以钉钉机器人的方式嵌入群聊，实现 CRM 场景的 @机器人 问答。

▸ 方案
  • 利用 BI 的 CLI 接口作为数据通道
  • 通过 Qoderwork Agent 封装为对话服务
  • 钉钉机器人 + IM 群聊模式，@ 触发问答
  • 首批落地场景：CRM 数据查询
▸ 踩坑
  • 数据范围受限：只能查询 BI 上已有的数据集
  • CLI 稳定性依赖 BI 平台，存在接口变动风险
  • 场景单一：仅覆盖 CRM，难以扩展到其他业务域
▸ 结论
  • 钉钉 + Agent 的 IM 模式验证成功
  • 打通了"用户 → 钉钉 → Agent → BI → 数据"全链路
  • 下一步：拓展到外卖等更多业务场景
🔴 阶段四

阿里云 Ontology — 连接 PG 数据库

拓展外卖场景时发现外卖数据存储在阿里云 PostgreSQL 中，Agent 无法直连。通过阿里云 Ontology 能力连接 PG，封装为知识图谱并发布为 Skill，实现业务指标规则化查询。

▸ 方案
  • 阿里云 Ontology 连接 PG 数据库
  • 将数据表结构封装为知识图谱（语义层）
  • 打包发布为 Skill，定义指标计算规则
  • 业务通过 Skill 中的指标规则提问
▸ 踩坑
  • 链路长：PG → Ontology → 知识图谱 → Skill → Agent
  • Ontology 建模成本高：需要人工定义实体关系与指标规则
  • 依赖阿里云生态：绑定了特定云厂商能力
▸ 结论
  • 突破了 Agent 无法直连 PG 的问题
  • 语义层（知识图谱）在复杂业务问数中确实有价值
  • 但链路过长，维护成本不可忽视
🟣 阶段五

PG → MCP 直连 — 简化数据通道

发现可以将 PostgreSQL 封装为 MCP（Model Context Protocol），直接在 Agent 中使用，简化了中间层链路。

▸ 方案
  • 将 PG 数据库封装为 MCP Server
  • Agent 通过 MCP 协议直连数据库
  • 减少 Ontology / Skill 中间层
▸ 优势
  • 路径更短：Agent → MCP → PG
  • 标准化协议：MCP 是行业通用标准
  • 灵活性更高：可自主控制查询逻辑
▸ 新问题
  • 缺少语义层：直接暴露数据库表结构给大模型，理解准确率下降
  • 安全风险：需要在 MCP 层做精细化的查询权限控制
  • 复杂指标：跨表聚合、业务口径仍需 Skill 层定义

🔴 当前待解决问题

基于 DataAgent 评测结果，梳理的 5 项仍需解决的核心问题。

01
知识管理

每个 Agent 需要独立的知识库

不能将所有业务知识都塞进 Skill 中。Agent 应当具备自己的知识库（Knowledge Base），用于存储业务术语定义、指标口径、常见问题答案等非结构化知识。Skill 应专注于可执行的操作定义，知识库应承担"业务字典"和"问答上下文"的角色。

  • Skill 膨胀：将业务知识硬编码到 Skill 中导致 Skill 臃肿、难以维护
  • 知识复用差：不同 Agent 之间的业务知识无法共享
  • 更新滞后：业务口径变化时，Skill 更新需要技术介入
02
多租户隔离

不同主题的 Agent 需要做权限与资源隔离

外卖、CRM、财务等不同业务主题需要独立的 Agent 实例。每个 Agent 之间需要做数据权限、Skill 能力、以及模型资源的隔离，确保业务数据不交叉泄露、计算资源不互相抢占。

  • 数据隔离：CRM Agent 不能访问外卖数据库，反之亦然
  • Skill 隔离：不同业务的指标规则、查询逻辑不能混用
  • 模型隔离：不同场景可能使用不同的模型（GPT / 开源模型），需要独立配置
  • 资源隔离：高并发场景下避免 Agent 间资源争抢
03
可观测性

用户问题追踪、Token 管理、溯源能力

当前缺少完整的问题生命周期追踪体系。需要记录每个用户问题的完整链路——从提问到 Agent 推理、到数据查询、到最终回答，以及每一步消耗的 Token。当回答出现偏差时，需要能快速定位到是哪个环节出了问题。

  • 问题追踪：记录"谁在什么时候问了什么问题，得到了什么回答"
  • Token 管理：按用户 / 部门 / 时间维度统计模型调用成本
  • 溯源：回答出错时，能追溯到是 SQL 写错、数据问题、还是大模型幻觉
  • 审计：满足企业合规审计需求
04
权限映射

数据权限与人员权限的对应关系

数据库中的行级数据权限（如区域经理只能看自己区域的数据）需要与 Agent 层的用户身份体系建立映射。当前缺少从"钉钉用户"到"数据库权限"的自动化映射机制。

  • 行级安全：不同用户查询同一指标时应返回不同范围的数据
  • 身份传递：钉钉用户身份需要透传到数据查询层
  • 动态权限：人员调岗、离职等场景下权限需实时生效
  • 权限审计：能够追溯"谁在什么时候访问了什么数据"
05
生态集成

与钉钉及企业生态的深度集成

钉钉是 Wagas 的一线业务入口，Agent 需要与钉钉深度集成——不仅是消息收发，还包括审批流、日程、待办、以及企业内部系统（ERP、HR 等）的打通。

  • 钉钉机器人消息体验优化：支持卡片消息、图文混排、快捷操作按钮
  • 审批流集成：问数结果可直接触发审批流程（如异常数据告警 → 发起整改工单）
  • 单点登录：统一钉钉身份认证，无需二次登录
  • 企业系统打通：ERP、HR、POS 等系统数据通过统一入口查询
